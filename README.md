## 核心亮点：Multi-Agent 并行调度架构 🤖

本项目采用 **4 个独立 Agent + 1 个协调器** 的多智能体架构，各 Agent 并行执行、多数据源交叉验证，最后由协调器汇聚合并，生成高质量出行方案。

### 各 Agent 职责 & 数据源

| Agent | 职责 | 数据源 | 亮点 |
|-------|------|--------|------|
| 🌤️ **天气情报 Agent** | 多源天气采集、交叉验证、出行气象建议 | 高德天气 API + 和风天气 API + 本地气候模型（基于郑州12个月份特征） | 三源交叉验证，**温差 ≤2°C 确认一致** 才输出高置信度结果；自动生成高温/降雨/大风预警 |
| 🏛️ **景点搜索 Agent** | 多源 POI 发现、场景匹配、智能排序 | 郑州本地景点数据库 + 高德 POI 搜索 + OSM Nominatim + 本地餐饮库 | 按 4 种出游场景匹配不同类别标签，**三源去重合并** 后按场景加权排序，支持雨天室内优先 |
| 🚗 **交通路线 Agent** | 路线规划、时间估算、出行方式建议 | 本地路线拓扑引擎（Haversine 公式 + 路网系数） + 高德驾车导航 | 按场景智能推荐交通方式（家人→自驾 / 朋友→拼车 / 恋人→打车 / 单人→地铁+单车）；含红绿灯数量 |
| ⭐ **评价口碑 Agent** | 多源口碑采集、评分汇总、人流量预测 | 本地评分数据库 + 高德 POI 评分 + 社媒热度模型 | 支持情感分类（极佳/推荐/尚可）；按周末/季节预测人流量；生成**四场景独立热门话题** |

### 协调器设计

- **并行执行**：4 个 Agent 通过 `Promise.all` 同时启动，不相互阻塞
- **实时状态反馈**：UI 实时展示每个 Agent 的执行进度条和状态（idle → working → done/error）
- **结果汇聚**：自动合并 4 路结果，提取天气/景点/交通/口碑数据，生成统一出行方案
- **容错降级**：单个 Agent 失败不影响其他，协调器自动跳过异常并继续

## 技术架构

```
┌────────────────────────────────────────────────────────────┐
│                      浏览器（前端）                          │
│  ┌──────────┐  ┌──────────────────────────────────────┐   │
│  │  UI 交互  │  │      Multi-Agent 引擎 (纯前端)        │   │
│  │  场景选择  │  │  Coordinator → 4 Agent 并行调度      │   │
│  │  地图渲染  │  │  结果汇聚 → 行程生成 → 实时渲染       │   │
│  │  拖拽排序  │  └────────────┬─────────────────────────┘   │
│  └──────────┘                │                             │
└──────────────────────────────┼─────────────────────────────┘
                               │ ① API 请求
                               ▼
                    ┌──────────────────┐
                    │   server.js      │
                    │   CORS 代理中转    │
                    │   (180 行极简后端) │
                    └──────┬───────────┘
                           │ ② 转发
                           ▼
           ┌───────────────────────────────┐
           │          外部 API 层           │
           │  AI模型 │ 高德 │ 和风 │ OSM │
           └───────────────────────────────┘
```

- **前端 Multi-Agent 引擎**：4 个 Agent 在浏览器侧并行执行，各自独立调用数据源，协调器汇聚结果
- **server.js**：180 行极简后端，仅做两件事——托管静态文件 + 转发 API 请求绕过浏览器 CORS
- **外部服务**：AI 大模型（行程生成/评价）、高德（天气/POI/路线）、和风（备用天气）、OSM Nominatim（免费地理编码）

## 快速启动

### 0. 克隆项目

```bash
git clone <repo-url>
cd chuxingzhushou
```

### 1. 配置 API Key



首次使用需要在页面内配置 API Key（页面左侧 **API 密钥配置** 按钮）：

- **高德地图 Key**：用于天气和地理编码（[申请地址](https://lbs.amap.com/)）
- **和风天气 Key**：用于备用天气数据源（[申请地址](https://dev.qweather.com/)，可选）
- **LLM API**：任何 OpenAI 兼容接口的模型均可使用，点击预设按钮一键填入主流厂商配置：<br>
  &nbsp;&nbsp;`DeepSeek` / `通义千问` / `OpenAI` / `智谱GLM` / `Kimi`<br>
  （只需填入 API URL + Key + 模型名称，详见各厂商文档）

> 🔒 **安全性说明**：所有 API Key 仅保存在你本地的浏览器 localStorage 中，**不会**写入任何代码文件，**不会**被 Git 追踪上传。每次打开页面 Key 会自动回填，无需反复输入。

### 2. 启动服务器

```bash
cd chuxingzhushou
node server.js
```

终端显示：
```
🚀 Server running at http://localhost:3000
📡 LLM proxy ready at /api/proxy/llm
```

### 3. 打开浏览器

访问 **http://localhost:3000**

### 4. 测试 API 连通性（可选）

```bash
node test_api.js
```

> ⚠️ `test_api.js` 需要你先编辑文件填入 API Key 才能运行。**请勿**将填了真实 Key 的版本提交到 Git。

## 常见问题

**Q: 为什么需要 server.js？**
A: 浏览器 CORS 策略阻止网页直接调用外部 API，server.js 作为本地代理转发请求。

**Q: API Key 安全吗？会被上传到 GitHub 吗？**
A: 安全。API Key 仅保存在你浏览器的 localStorage 中，不写入任何代码文件。项目已配置 `.gitignore`，测试脚本的 Key 默认为占位符。只要不在 `test_api.js` 里填入真实 Key 后提交，就不会泄露。

**Q: AI 模式失败怎么办？**
A: 会自动降级到默认模式。查看 F12 Console 的错误日志，通常原因是 API Key 无效或额度用完。

**Q: 地图 marker 重叠？**
A: 已处理——地理编码失败的 fallback 坐标会加随机偏移避免重叠。每个 marker 有独立编号和颜色，与图例一一对应。
